# Audit Corpus

This directory contains a small, committed corpus of ponytail audit outputs for
demo and benchmark workflows across `dr-providers`, `dr-queues`,
`dr-bottleneck`, and `dr-dspy`.

The current corpus was generated by running pi agent with `gpt-5.5` against this
repo using the prompt:

```text
please run a ponytail audit of this repo
```

The files are intended to be stable inputs. Generated model outputs, parsed
results, database exports, and benchmark reports should be written elsewhere
unless they become curated benchmark snapshots.

I was thinking that we could build up to an llm in the loop review normalizing workflow that used really really cheap llms to try to do specific tasks related to parsing and normalizing theses logs and then if we had the script always write to the same mongo then over time we'll get more and more samples on this same problem which eventually will give a fun dataset about how performance changes, how accurate the mean estimate is with few vs many samples, etc.

## dr-providers
I was thinking the demo in this repo could be choosing a small set of super cheap models, a small set of tasks, and then running each model on each input for each task. Perhaps tasks like (1) extract the stats included in each file into a consistent json format (2) normalzie the reported lines into a json format, (3) deduplicate across all the audits into a fixed set of suggestions, (4) identify which thinking level produced the most "shrink" suggestions (5) identify which category had the highest average cost and report the cost, which assumes we nromalize the category names to not be (off throught xhigh).

I think it will be interesting to see which tasks the models can do perfectly and which they can't do at all.

## dr-queues

We can test out interesting queue shapes by doing a fully programatic sequence of operations on the audit files and verify that in many different configurations and in fairly large multi-piece systems the queue impls still work and are convenient primitive designs to use.

## dr-bottleneck

Then we can start chaining these mini tasks together. This would make a great way to test out the flexibility of the queue design.  so like I could imagine trying to build a workflow where there are to initial nodes that both got the same audit text but one normalized jsut the review commetns into json, the other normalized just the stats into json and they were both supposed to preserve an id in the same way in the output.  and they both dump their output in the same queue.  then there was a compute step where programmatic parsing happens where say when the worker receives the review commennt json it does some parsing and then inserts it into the db under the key.  and when it receives the stats json it does some parsing and then inserts it into the db under the key.  but in both cases, when the stats and comments are both in the db then the worker pairs them and adds them to the ouptut queue.  tests flexibility and demos a real workflow

## Suggested Updates

Treat this corpus as the shared input fixture, not as the final benchmark
result. The useful benchmark data should come from rerunning focused tasks over
these files and storing every attempt with enough metadata to compare models,
prompts, parsers, and workflow designs over time.

Recommended first tasks for `dr-providers`:

- Extract run-header stats into a consistent JSON shape.
- Normalize each audit line into structured JSON with category, suggestion,
  referenced files, replacement, and rationale.
- Deduplicate audit lines into a fixed set of suggestions.
- Answer aggregate questions from normalized data, such as which thinking level
  produced the most `shrink` suggestions.
- Normalize level/category names and compute cost summaries.

Recommended storage shape for generated runs:

- Stable `audit_id` derived from the corpus path.
- Input checksum.
- Model id and provider.
- Prompt/task version.
- Raw response.
- Parsed response.
- Validation errors.
- Deterministic score when a gold answer exists.

Recommended repo roles:

- `dr-providers`: run model/task/input matrices and record raw plus parsed
  outputs.
- `dr-queues`: exercise queue shapes over fixed audit-file work units.
- `dr-bottleneck`: demonstrate fanout/fanin workflows, partial-result joins,
  retries, persistence, and output queues.
- `dr-dspy`: optimize one narrow subtask from the workflow rather than the whole
  pipeline.

Keep model selection configurable. Cheap model availability and pricing change,
so runs should capture the selected model ids and price snapshot instead of
assuming that the committed corpus determines the model set.
